What it reads out of your code

Five layers of understanding, generated from the same parse pass.

[ 01 ]

Block diagrams

Module hierarchy rendered as a navigable schematic — instantiations, port connections, and clock domain boundaries drawn automatically.

[ 02 ]

Module descriptions

Plain-language summaries of what each module does, inferred from logic structure, naming, and surrounding context — not just comments.

[ 03 ]

Signal explanations

Every port annotated with direction, width, and a sentence on what it controls or reports — built for the reviewer who didn't write the code.

[ 04 ]

Data flow

Traces how a value moves from input pin to output pin across module boundaries, flagging where it's registered, gated, or transformed.

[ 05 ]

Architecture documentation

A written architecture doc assembled from the above — exportable as Markdown, ready to drop into your repo or design review deck.

[ 06 ]

Spec compliance diffing

Upload the spec PDF alongside your RTL and get a structured list of unimplemented requirements, parameter mismatches, and missing error paths.

Under the hood

Four steps from raw RTL to a reviewable document.

[ 01 ]

Parse

Module boundaries, port lists, parameters, and instantiation tree extracted from the raw source text.

[ 02 ]

Reason

Each module's logic is read in context — always-blocks, state machines, and signal naming inform what it actually does.

[ 03 ]

Cross-reference

If a spec is provided, every requirement is checked against the parsed RTL for a matching implementation.

[ 04 ]

Render

Block diagrams, module docs, and findings are assembled into the workbench view and exportable Markdown.
